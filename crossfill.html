The Problem

Crossfill had data. It didn't have direction.

The platform surfaced analytics beautifully — but when teams wanted to act on insights, they had to leave the product entirely. There was no structured way to create, assign, or track optimization actions. Teams were stitching together spreadsheets, Notion docs, Jira tickets, and Slack threads just to move from "we see the problem" to "we did something about it."

The result: brilliant insights that went nowhere. A product that felt complete but wasn't. My job was to design the missing layer — the Actions workflow.

My Role

From research to shipped product — I led every step.

As the Actions page lead, I owned the full design process: user research, workflow mapping, concept exploration, prototyping, and handoff. I worked closely with the PM and two engineers to move from zero to a shipped feature in one quarter.

Understanding the Workflow

Two teams. One product. Completely different mental models.

Research interviews revealed a deep tension between how engineering and customer success teams thought about "taking action" on data.

Engineering Mindset

Structured. Ticket-driven. Async.

  • Wants tasks with clear scope and acceptance criteria
  • Lives in sprint cycles — prefers batched work
  • Thinks in systems: dependencies, blockers, priorities
  • Needs integration with existing dev tooling
Customer Success Mindset

Relational. Outcome-driven. Reactive.

  • Wants to act on a customer insight immediately
  • Thinks in goals, not tasks — "fix the churn problem"
  • Needs visibility across clients, not code repos
  • Wants a single place to see "what's being done"
Design Principle

"The Actions workflow had to feel native to how each team worked — not force either group to adopt the other's mental model. Flexibility without chaos."

Early Exploration

Two directions. One clear winner.

I explored two fundamentally different approaches before narrowing down. Each had a distinct philosophy about where actions should live.

↑ Chosen Direction

Dedicated Actions Page

A purpose-built home for all optimization actions — separate from the analytics views, with its own navigation, filtering, and workflow tools. Actions become a first-class feature, not a sidebar panel.

Why this won: Gave the feature the visual weight it deserved. Created a mental separation between "looking at data" and "acting on data." Scalable as the feature grew.
↓ Discarded

Contextual Inline Actions

Actions embedded directly within report and metric views — clicking an insight immediately spawns a task creation flow without leaving the current context.

Why this lost: Created fragmentation. There was no single place to see "all the work being done." Power users managing multiple actions across multiple reports got lost quickly.
Design Structure

A four-state flow for every action.

Every optimization action moves through the same four lifecycle states, giving teams a shared vocabulary for progress.

01
Identify
Spot the insight, create the action with context attached
02
Assign
Set owner, due date, category, and priority
03
Execute
Work happens; status updates keep stakeholders aligned
04
Measure
Close the loop — did the action move the metric?
Iteration 1

Getting the structure right.

The first round of testing surfaced two clear problems with the initial layout — both fixable with focused redesigns.

Problem Found
Table view overwhelmed users
Dense rows with too many columns made it hard to scan. Users said it "felt like a spreadsheet" — not in a good way.
  • Reduced visible columns to 5 (from 9)
  • Added color-coded status pills for instant scanning
  • Introduced a card-list toggle for different work styles
  • Progressive disclosure: full detail on click/expand
Problem Found
Action creation felt disconnected
The modal for creating a new action had no context about which insight triggered it. Users lost the "why" behind the action.
  • Added "linked insight" field — attaches the source metric
  • Auto-populated action title from insight description
  • Kept modal slim — 3 required fields only, rest optional
  • Inline preview of linked chart in creation flow
Iteration 2

The inline editor that changed everything.

After the first round of revisions, a new pattern emerged from power users: they wanted to edit action details without leaving the list. Navigating to a detail page for a quick status change felt like too much friction.

The solution was an expandable inline editor — click any row to reveal an edit panel within the table itself. No navigation, no lost context, no modal interruptions.

Inline Editor

Edit-in-place, stay in flow

An expandable row panel that surfaces the most-edited fields without a full page change.

  • Status update with one click — no modal
  • Owner reassignment via inline people picker
  • Notes/comments thread within the row
  • Linked insight remains visible for reference
  • Keyboard shortcut (E) to toggle editor on focused row
Impact

A reporting tool became an actionable product.

Shipping the Actions page changed how Crossfill users engaged with the platform — and how the team talked about the product itself.

🔗
4 tools replaced by one workflow
Teams stopped switching between spreadsheets, Notion, Jira, and Slack. The entire optimization loop now lives in Crossfill.

Reflection

Three things I carry forward.

1
Learning
0→1 work demands a stronger opinion upfront
When there's no existing pattern to improve, the designer has to make bold structural decisions early. Staying neutral for too long creates drift. I learned to commit to a direction, articulate why, and invite critique of that specific choice — not an open-ended "what do you think?"
2
Learning
Friction often hides in transitions, not features
The biggest UX wins came not from adding features, but from smoothing the moments between them. The inline editor wasn't a new feature — it was a better transition between "viewing" and "editing." I now map every workflow as a series of transitions first.
3
Learning
Ship something learnable, not just something finished
We could have waited for a "complete" Actions feature. Instead, we shipped a focused MVP that taught us what users actually did — which turned out to be quite different from what we assumed. The best design decisions in this project came after shipping, not before.
